构建并分享项目¶

这份教程会把已在编辑器中验证的项目变成 Windows 独立构建，当前针对 0.2.1 预览版构建流程。

预计时间： 构建依赖可用后约 10–20 分钟。

开始之前¶

  • 项目能够打开，Console 不反复输出异常。
  • 至少有一个场景已保存到项目中。
  • 场景能够在 Play 模式正常运行。
  • 有一个空间充足、可写的输出目录。

1. 打开 Build Settings¶

打开编辑器的 Build Settings 窗口。它负责有序场景列表与独立播放器配置。

把当前打开的场景加入列表。未保存的场景无法加入，因为构建需要稳定的项目相对路径。把运行时可能加载的所有场景都加入列表，并主动安排顺序：构建索引 0 是启动场景。

检查点¶

场景列表至少包含一个有效的已保存场景，并且没有路径缺失警告。

2. 配置播放器¶

第一次构建需要设置：

  • 游戏名称： 可执行文件和面向用户的产品名。
  • 输出目录： 独立、空白或可随时清理的构建目录。
  • 显示模式： 窗口或适合项目的全屏模式。
  • 窗口大小与可调整状态： 首次运行时的窗口行为。
  • 图标与启动画面： 第一次验证构建可以暂时省略。
  • Debug / JIT / LTO： 基础构建成功前保持预览版默认设置。

不要把项目目录本身选作输出目录。构建流程会创建并清理生成文件；分离输出能保护项目资源，也能清楚展示真正需要分发的内容。

3. 执行构建¶

选择 Build 只生成独立产物，或选择 Build and Run 在打包后由编辑器启动结果。

流水线会验证项目与场景列表、编译播放器、复制资源、处理启动内容、写入构建清单并完成最终清理。构建停止时优先处理第一条错误；后续消息可能只是失败后的清理结果。

预期结果¶

Build Settings 报告成功并给出最终输出目录。目录中包含可执行文件及其需要的运行时文件。

4. 在编辑器之外测试¶

关闭或最小化编辑器，直接从输出目录启动可执行文件。

逐项验证：

  • 启动场景正确；
  • 编辑器不占用焦点时输入仍然工作；
  • 场景切换能加载 Build Settings 中的全部场景；
  • 纹理、音频、脚本与其他必要资源都存在；
  • 窗口大小、全屏行为、标题与图标符合配置；
  • 干净启动不依赖输出目录之外的文件。

只测试 Build and Run 不够。可分享构建还必须能从打包目录直接运行。

5. 制作分享压缩包¶

退出游戏，压缩完整输出目录，不要只复制 .exe。保持目录结构不变。发布前：

  1. 把压缩包解压到另一个本地目录；
  2. 启动解压后的副本；
  3. 测试启动场景与至少一次场景切换；
  4. 在发布说明中注明引擎版本与已知预览限制。

常见失败¶

Build 按钮不可用¶

  • 保存当前场景。
  • 至少把一个场景加入 Build Settings。
  • 选择有效且可写的输出目录。

运行时打开了错误场景¶

把目标启动场景移动到构建索引 0，保存 Build Settings 后重新构建。

资源在编辑器中正常，但构建后缺失¶

  • 确认资源位于项目内部，并使用项目相对引用。
  • 阅读最早出现的资源复制或加载错误。
  • 输出到干净目录，避免旧文件干扰判断。

可执行文件只能在开发机器上运行¶

在干净的 Windows 环境测试完整输出目录。不要从运行时目录中单独拿走可执行文件分发。

完成标准¶

把压缩包重新解压后，不启动编辑器也能运行；启动场景、输入、资源加载和退出都正常，才算完成。
